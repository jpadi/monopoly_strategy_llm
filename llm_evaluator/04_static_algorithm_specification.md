# Static Algorithm Evidence Specification

Version: 1.0

## Purpose

This document defines the deterministic evidence that a Static Algorithm may generate for the Monopoly Trade Evaluation System.

It defines the evidence contract only.

It does not prescribe formulas, weights, or execution order.

A concrete deterministic algorithm implementing this contract is specified in:

**../static_evaluator/static_algorithm_specification.md**

The Static Algorithm receives the same structured input defined in:

**03_input_schema.md**

The Static Algorithm must operate using the current input model:

- `metadata`
- `game_configuration`
- `board_state_before`
- `trade`
- `board_state_after`
- `static_analysis_control`
- `static_algorithm_evidence`
- `player_profiles`
- `decision_pattern_profiles`

The Static Algorithm should primarily analyze:

- `game_configuration`
- `board_state_before`
- `trade`
- `board_state_after`

The Static Algorithm may ignore `player_profiles` and `decision_pattern_profiles` unless a specific algorithm version explicitly supports them.

The Static Algorithm must ignore `static_analysis_control`; that section controls how the Judge obtains and consumes evidence, not how the algorithm calculates it.

Its role is to generate deterministic evidence.

It does not replace the LLM evaluator.

It does not determine player intent.

It does not perform Decision Pattern Analysis.

Its purpose is to produce objective calculations that help the evaluator understand the competitive impact of the trade.

---

## Evidence Producers

This evidence contract applies identically to every producer of Static Algorithm Evidence:

- deterministic software implementing a Static Algorithm Specification;
- the LLM Judge, when it executes a Static Algorithm Specification itself under the LLM Static Fallback rules defined by the Judge Specification (`01_judge.md`).

Judge-generated fallback evidence must use the same structure, the same field names, the same status values, and the same traceability rules defined by this document.

No producer may substitute a prose summary for the canonical evidence structure.

The provenance of the evidence — supplied by deterministic software or generated by the Judge — is recorded in the Judge's output (`static_analysis_result.evidence_source`), not inside this evidence object.

---

## Core Principle

The Static Algorithm should expose both:

1. Intermediate calculations.
2. Final conclusions.

Providing only a final score is discouraged.

Every important final conclusion should be traceable to one or more intermediate calculations.

The evaluator should be able to understand:

- what was calculated (`calculation_type`, `description`);
- which input data was used (`subject`, `input_references`, `input_values`, `sources`);
- how the value was produced (`formula`, `procedure`, `intermediate_values`);
- how the calculation affected the final deterministic conclusion (`dependent_calculation_ids` and the conclusion's item references).

These parenthesized names refer to the canonical Intermediate Calculation Item structure defined below.

---

## Input Alignment

The Static Algorithm must use the same entity identifiers defined in the input schema.

Examples include:

- Player identifiers from `board_state_before.players`
- Property identifiers from `board_state_before.properties`
- Tradable Asset identifiers from `board_state_before.tradable_assets`
- Trade identifiers from `trade.id`
- Transfer identifiers from `trade.transfers`

All Static Algorithm Evidence should reference these identifiers whenever possible.

The Static Algorithm must not create unrelated identifiers for existing players, properties, assets, or trades.

This ensures that the evaluator can trace every calculation back to the original input.

---

## Required Analysis Scope

At minimum, a Static Algorithm should be able to compare:

- the board state before the trade;
- the trade itself;
- the board state after the trade.

The algorithm should identify deterministic changes caused by the trade, including:

- cash changes;
- property ownership changes;
- tradable asset changes;
- monopoly creation;
- monopoly loss;
- blocking changes;
- mortgage changes;
- liquidity changes;
- development opportunities;
- house supply changes;
- hotel supply changes;
- bankruptcy risk changes;
- competitive advantage changes.

The Static Algorithm should not rely on undocumented assumptions when required values are missing.

If a calculation cannot be completed because required input data is missing, the algorithm should record that limitation in its evidence.

## Evidence Object Structure

All deterministic evidence generated by the Static Algorithm should follow a consistent structure.

The goal is to make every calculation traceable, inspectable, and useful to the evaluator.

A Static Algorithm Evidence object should be compatible with the `static_algorithm_evidence` section defined in `03_input_schema.md`.

---

### Evidence Structure

```json
{
  "id": "",
  "algorithm_id": "",
  "algorithm_name": "",
  "algorithm_version": "",
  "purpose": "",
  "intermediate_calculations": [],
  "final_conclusions": {},
  "metadata": {},
  "statistics": {}
}
```

---

### Algorithm Identity

- `algorithm_id` — stable machine-readable identifier of the producing algorithm. It must equal the `specification_id` declared by the algorithm's specification (for the concrete algorithm in this repository: `monopoly_static_algorithm`).
- `algorithm_name` — human-readable name of the producing algorithm.
- `algorithm_version` — the `specification_version` implemented by the producer.

These fields allow the Judge to validate supplied evidence against `static_analysis_control.required_algorithm` and to report the algorithm identity in its output.

---

### intermediate_calculations

The `intermediate_calculations` array contains the detailed deterministic calculations produced by the algorithm.

Each calculation explains one specific measurable result.

Examples:

- cash change calculation;
- property value calculation;
- mortgage capacity calculation;
- liquidity calculation;
- monopoly completion calculation;
- development potential calculation;
- house control calculation;
- railroad value calculation;
- trade ratio calculation.

Each intermediate calculation must be understandable and independently reproducible without needing to rerun the algorithm.

Every element of the array is one canonical **Intermediate Calculation Item**, defined below. This envelope is shared by every Static Algorithm Evidence producer.

---

### Intermediate Calculation Item

```json
{
  "id": "",
  "calculation_type": "",
  "description": "",
  "status": "COMPLETE",
  "side": "before",
  "subject": {
    "player_ids": [],
    "property_ids": [],
    "group_ids": [],
    "trade_id": null,
    "transfer_ids": []
  },
  "input_references": [],
  "input_values": {},
  "sources": {},
  "formula": null,
  "procedure": null,
  "intermediate_values": {},
  "result": null,
  "unit": null,
  "limitations": [],
  "missing_inputs": [],
  "dependent_calculation_ids": [],
  "metadata": {}
}
```

#### id

A unique and stable identifier for the calculation item within the Static Algorithm Evidence object.

Final conclusions, labels, warnings, bonuses, penalties and scores should reference these identifiers whenever applicable.

#### calculation_type

A stable machine-readable identifier defined by the producing algorithm's Evidence Output Mapping registry (for the Static Trade Imbalance Evaluator, the registry in `../static_evaluator/static_algorithm_specification.md`).

Examples include:

```text
base_asset_value
mortgage_capacity
available_construction_budget
monopoly_value
house_control_bonus
house_denial_bonus
railroad_value
blocking_value
strategic_opportunity_score
available_risk_resources
landing_exposure
repair_exposure
largest_threat
risk_margin
coverage_ratio
immediate_risk
development_capability
dangerous_monopoly_availability
strategic_value
trade_ratio
final_static_classification
```

Implementations must not invent synonymous identifiers for calculation types already defined by the registry.

#### description

A concise human-readable explanation of what was calculated.

#### status

Use exactly one of:

```text
COMPLETE
PARTIAL
UNAVAILABLE
ERROR
```

- `COMPLETE`: every required input was available and the calculation completed successfully.
- `PARTIAL`: part of the calculation completed, but one or more missing inputs reduced completeness.
- `UNAVAILABLE`: the calculation could not produce a meaningful result because required information was unavailable.
- `ERROR`: the calculation failed because the supplied input was inconsistent, invalid or could not be processed.

#### side

Use exactly one of:

```text
before
after
delta
trade
```

- `before`: calculation for `board_state_before`.
- `after`: calculation for `board_state_after`.
- `delta`: `after - before` contribution.
- `trade`: calculation that does not belong to only one board-state side.

#### subject

Contains stable identifiers for every entity materially involved in the calculation.

Arrays are used because a calculation may involve multiple players, properties, groups or transfers.

#### input_references

Machine-readable references to the exact input objects or fields used.

Examples:

```text
board_state_before.players[player_1].cash
board_state_after.properties[property_boardwalk].rent_table.hotel
game_configuration.group_multipliers[orange]
```

#### input_values

Named factual values read from the referenced input.

#### sources

Records the provenance of configurable or default values.

Examples include:

```text
GAME_CONFIGURATION_OVERRIDE
BOARD_STATE_RENT_TABLE
CLASSIC_DEFAULT
NEUTRAL_CUSTOM_GROUP_DEFAULT
RISK_REFERENCE_DEFAULT
SPECIFICATION_DEFAULT
IMPLEMENTATION_OVERRIDE
```

The object may contain multiple named source entries.

#### formula

The mathematical formula used when the calculation is formula-based.

#### procedure

The deterministic procedure used when the result cannot be represented adequately by one formula.

At least one of `formula` or `procedure` should be present for every completed material calculation.

#### intermediate_values

Every intermediate value necessary to independently reproduce the result.

Intermediate arithmetic must not be hidden inside prose.

#### result

The calculation result: number, boolean, string, array, structured object, or `null` when unavailable.

#### unit

Optional unit or semantic type.

Examples:

```text
currency
ratio
percentage
houses
hotels
development_level
boolean
classification
```

#### limitations

Known limitations affecting interpretation of the result.

#### missing_inputs

Required inputs that were unavailable.

This field is mandatory when status is `PARTIAL` or `UNAVAILABLE`.

#### dependent_calculation_ids

Identifiers of calculations that consume this result or whose reliability is affected by it.

#### metadata

Optional implementation-specific information that does not alter the calculation meaning.

Unknown metadata must not be required to understand or reproduce the result.

---

### Minimum Required Fields

Every item must contain:

```text
id
calculation_type
description
status
side
subject
input_references
input_values
sources
intermediate_values
result
limitations
missing_inputs
```

For `COMPLETE` calculations:

- `result` must not be null;
- `missing_inputs` must be empty;
- at least one of `formula` or `procedure` must be present for every material calculation.

For `PARTIAL` calculations:

- available partial results may be included;
- missing inputs and their impact must be explicit.

For `UNAVAILABLE` calculations:

- `result` must normally be null;
- the attempted calculation must still be published;
- `missing_inputs` and `limitations` must explain why no result exists.

For `ERROR` calculations:

- the error condition must be identified in `limitations`;
- no fabricated result may be published.

---

### final_conclusions

The `final_conclusions` object should summarize the deterministic result of the algorithm.

Examples:

- total value gained by each player;
- total value lost by each player;
- trade imbalance ratio;
- deterministic advantage estimate;
- generated static flags;
- major deterministic warnings.

Every material final conclusion — labels, warnings, bonuses, penalties, deterministic scores, review requirements, and the final static classification — must reference the `id` values of the supporting intermediate calculation items.

No final conclusion should depend on unreferenced hidden arithmetic.

---

### metadata (evidence object)

The evidence object's top-level `metadata` may contain implementation-specific information.

It must declare every Implementation Override applied to a Specification Default Value, including for each override:

- parameter name;
- specification default value;
- overridden value;
- optional reason.

An implementation using only Specification Default Values publishes an empty override declaration.

It must also declare the version of every specification dependency used by the producing algorithm — for the concrete algorithm in this repository, the Risk Reference (`monopoly_risk_reference` and its `specification_version`).

---

### Traceability Requirement

Every important calculation must reference the input objects used to produce it, through the item's `subject` and `input_references` fields.

Examples:

- `player_id`
- `property_id`
- `trade_id`
- `transfer_id`
- `board_state_before`
- `board_state_after`

The evaluator should be able to connect each deterministic result back to the original input.

---

### Missing Data

If a calculation cannot be completed because required data is unavailable, the Static Algorithm must record that limitation using the `PARTIAL` or `UNAVAILABLE` status of the Intermediate Calculation Item.

The algorithm should not invent missing values.

The item must make clear, through `description`, `missing_inputs`, and `limitations`:

- what calculation was attempted;
- what information was missing;
- how the missing information affected the result.

---

### No Strategic Intent

Static Algorithm Evidence must not include conclusions about player intent.

It must not claim that a player intentionally performed:

- collusion;
- kingmaking;
- trolling;
- griefing;
- cheating.

The Static Algorithm may identify deterministic imbalance.

The LLM evaluator remains responsible for final interpretation and reporting.

## Calculation Categories

The Static Algorithm may generate many different deterministic calculations.

Each calculation belongs to a well-defined category.

Categories organize deterministic evidence by the Monopoly concept being evaluated rather than by the implementation that produced it.

Different Static Algorithms may implement the same category using different methods.

The evaluator should focus on the resulting evidence rather than the implementation details.

---

### Property Evaluation

Calculations related to the competitive value of individual properties.

Examples include:

- intrinsic property value;
- current property value;
- development value;
- monopoly contribution;
- blocking contribution;
- strategic importance.

---

### Financial Evaluation

Calculations describing the financial position of players.

Examples include:

- cash;
- liquidity;
- mortgage capacity;
- available development capital;
- recoverable assets;
- financial pressure.

---

### Monopoly Evaluation

Calculations related to complete and incomplete property groups.

Examples include:

- completed monopolies;
- near monopolies;
- monopoly completion value;
- monopoly loss;
- blocking value;
- development readiness.

---

### Development Evaluation

Calculations related to houses and hotels.

Examples include:

- immediate building potential;
- future building potential;
- available houses;
- available hotels;
- development efficiency;
- house shortage impact.

---

### Board Control Evaluation

Calculations describing control over important areas of the board.

Examples include:

- railroad control;
- utility control;
- house control;
- hotel control;
- blocking control;
- strategic board influence.

---

### Trade Evaluation

Calculations directly related to the evaluated trade.

Examples include:

- player gains;
- player losses;
- trade balance;
- trade ratio;
- generated bonuses;
- generated penalties;
- deterministic warnings.

---

### Competitive Evaluation

Calculations describing the overall competitive impact of the trade.

Examples include:

- competitive advantage changes;
- competitive disadvantage changes;
- bankruptcy risk;
- negotiation leverage;
- strategic opportunities created;
- strategic opportunities lost.

---

### Custom Categories

Static Algorithms may generate additional calculation categories.

Custom categories should be clearly documented.

Whenever possible, new categories should complement existing evidence rather than duplicate it.

## Calculation Principles

Every deterministic calculation produced by a Static Algorithm should follow a common set of principles.

These principles ensure that calculations remain transparent, reproducible, and useful to the evaluator.

Different algorithms may use different mathematical models.

However, every calculation should satisfy the same fundamental requirements.

---

### Deterministic

A calculation should always produce the same result when given the same input.

The algorithm should never introduce randomness.

Two identical Board States and Trades should always generate identical deterministic evidence.

---

### Traceable

Every calculation should be traceable back to the original input.

The evaluator should be able to determine:

- which input objects were used;
- which values were read;
- which intermediate values were produced;
- how the final result was obtained.

Every important conclusion should be explainable.

---

### Explainable

Calculations should prioritize transparency over mathematical complexity.

Whenever practical, expose intermediate values instead of only final results.

The evaluator should be able to understand the reasoning without reading the algorithm's source code.

---

### Independent

Each calculation should represent one specific Monopoly concept.

Examples include:

- liquidity;
- monopoly completion;
- mortgage capacity;
- blocking value.

A calculation should avoid combining multiple unrelated concepts into a single value whenever those concepts can be evaluated independently.

---

### Reusable

The result of one calculation may be used as input for another calculation.

Examples include:

- Property Evaluation contributing to Player Evaluation;
- Player Evaluation contributing to Trade Evaluation;
- Liquidity contributing to Bankruptcy Risk;
- Monopoly Evaluation contributing to Development Evaluation.

Whenever calculations depend on previous calculations, those dependencies should remain explicit.

---

### Modular

Each calculation should be independently replaceable.

Improving one calculation should not require redesigning unrelated calculations.

This allows different algorithm versions to improve individual concepts while preserving compatibility with the rest of the evaluation pipeline.

---

### Comparable

Whenever possible, calculations should produce results that can be compared before and after the trade.

Examples include:

- liquidity before vs after;
- monopoly count before vs after;
- development potential before vs after;
- player valuation before vs after.

The Static Algorithm should focus on identifying deterministic changes produced by the trade.

---

### Objective

Calculations should evaluate only measurable Monopoly concepts.

They should never attempt to infer:

- player intent;
- player psychology;
- decision quality;
- collusion;
- kingmaking;
- sportsmanship.

Those concepts belong exclusively to the LLM evaluator.

## Core Calculation Types

This section defines the main calculation types that a Static Algorithm may generate.

Each calculation type should produce deterministic evidence that helps the evaluator understand the competitive impact of a trade.

These calculation types are not mandatory for every Static Algorithm.

A simple Static Algorithm may generate only a subset.

A more advanced Static Algorithm may generate many or all of them.

---

### Cash Change Calculation

Measures how much cash each player gains or loses as a direct result of the trade.

This calculation should compare cash before and after the trade.

It should identify:

- cash given;
- cash received;
- net cash change;
- post-trade cash;
- whether the player remains liquid after the trade.

---

### Property Ownership Change Calculation

Measures which properties changed ownership.

This calculation should identify:

- properties gained;
- properties lost;
- property groups affected;
- whether ownership changes create or destroy monopolies;
- whether ownership changes create or remove blocks.

---

### Liquidity Calculation

Measures each player's ability to generate usable cash.

Liquidity may include:

- current cash;
- mortgage capacity;
- recoverable value from selling houses;
- recoverable value from selling hotels;
- other immediately convertible resources allowed by the Game Configuration.

This calculation should be especially important when a player accepts a trade that leaves them with low cash.

---

### Mortgage Capacity Calculation

Measures how much cash a player can generate by mortgaging currently unmortgaged properties.

This calculation should identify:

- unmortgaged properties;
- mortgage value of each property;
- total mortgage capacity;
- mortgage capacity before the trade;
- mortgage capacity after the trade.

---

### Monopoly Completion Calculation

Measures whether the trade creates, destroys, or changes control of complete property groups.

This calculation should identify:

- monopolies completed;
- monopolies lost;
- monopolies transferred;
- monopolies made possible;
- monopolies blocked.

---

### Development Potential Calculation

Measures whether a player can immediately build houses or hotels after the trade.

This calculation should consider:

- complete monopolies;
- available cash;
- available houses;
- available hotels;
- building rules;
- mortgage status;
- even-building requirements.

A monopoly that cannot be developed immediately may have much lower immediate danger than one that can be developed at once.

---

### House Control Calculation

Measures the effect of the trade on house supply and house control.

This calculation should identify:

- houses currently on the board;
- houses remaining in the bank;
- houses a player can immediately buy;
- whether the trade enables house scarcity;
- whether the trade prevents opponents from building.

---

### Railroad Evaluation

Measures the competitive value of railroad ownership.

This calculation should identify:

- railroad count before and after the trade;
- whether a player gains or loses railroad control;
- expected cash-flow relevance;
- blocking or negotiation value of railroad ownership.

---

### Utility Evaluation

Measures the competitive value of utility ownership.

This calculation should identify:

- utility count before and after the trade;
- whether both utilities become controlled by one player;
- expected rent threat;
- situational importance based on current board state.

---

### Blocking Value Calculation

Measures whether the trade creates, preserves, removes, or weakens blocks.

This calculation should identify:

- groups blocked before the trade;
- groups blocked after the trade;
- players affected by each block;
- whether the trade opens a dangerous monopoly;
- whether the trade prevents an opponent from completing a monopoly.

---

### Bankruptcy Risk Calculation

Measures whether the trade increases or decreases immediate survival risk.

This calculation should consider:

- post-trade cash;
- liquidity;
- nearby rent threats;
- mandatory mortgage transfer costs;
- existing rent obligations;
- immediate development by opponents;
- tax or card risks if available.

---

### Trade Ratio Calculation

Measures the deterministic value exchanged by each side.

This calculation may compare:

- cash value;
- property value;
- mortgage value;
- strategic value;
- development value;
- blocking value;
- liquidity impact.

The ratio should be transparent and traceable to its components.

---

### Competitive Advantage Calculation

Measures the overall deterministic competitive change caused by the trade.

This calculation may combine multiple lower-level calculations.

It should clearly expose:

- which components were included;
- how each component contributed;
- which player gained advantage;
- which player lost advantage;
- the final deterministic conclusion.

This calculation should never include player intent or Decision Pattern Analysis.

## Calculation Types

Static Algorithm calculations represent deterministic evidence generated by the algorithm.

Each calculation should evaluate one well-defined Monopoly concept.

A calculation should produce objective, reproducible, and traceable evidence.

The evaluator should use these calculations as supporting evidence rather than final conclusions.

This document defines the types of calculation evidence that may be provided to the evaluator.

It does not define how those calculations are implemented.

The implementation details belong to the corresponding Static Algorithm documentation.

---

### Common Calculation Categories

Static Algorithm Evidence may include calculations from categories such as:

- Property Evaluation
- Financial Evaluation
- Monopoly Evaluation
- Development Evaluation
- Board Control Evaluation
- Trade Evaluation
- Competitive Evaluation

These categories organize deterministic evidence.

They do not prescribe formulas, weights, execution order, or implementation details.

---

### Property Evaluation

Property Evaluation calculations describe deterministic information related to individual properties.

Examples include:

- property value;
- mortgage value;
- development status;
- rent potential;
- group membership;
- ownership changes.

---

### Financial Evaluation

Financial Evaluation calculations describe deterministic information related to player finances.

Examples include:

- cash;
- liquidity;
- mortgage capacity;
- recoverable value from developments;
- post-trade financial position;
- immediate payment pressure.

---

### Monopoly Evaluation

Monopoly Evaluation calculations describe deterministic information related to property groups.

Examples include:

- completed monopolies;
- incomplete monopolies;
- monopolies created by the trade;
- monopolies lost by the trade;
- monopoly blocking status.

---

### Development Evaluation

Development Evaluation calculations describe deterministic information related to houses and hotels.

Examples include:

- immediate building potential;
- available houses;
- available hotels;
- current development level;
- development cost;
- development constraints.

---

### Board Control Evaluation

Board Control Evaluation calculations describe deterministic information related to control over important board resources.

Examples include:

- railroad count;
- utility count;
- house control;
- hotel control;
- blocking control;
- property group influence.

---

### Trade Evaluation

Trade Evaluation calculations describe deterministic information directly related to the evaluated trade.

Examples include:

- assets given;
- assets received;
- cash exchanged;
- property ownership changes;
- trade ratio;
- direct value exchanged.

---

### Competitive Evaluation

Competitive Evaluation calculations describe deterministic information about the overall competitive impact of the trade.

Examples include:

- advantage gained;
- disadvantage created;
- risk increased;
- risk reduced;
- strategic opportunity created;
- strategic opportunity lost.

These calculations may summarize multiple lower-level calculations.

They should still remain traceable to their supporting evidence.

## Evidence Consumption

The evaluator should treat every Static Algorithm calculation as deterministic supporting evidence.

Static Algorithm calculations provide objective analysis of the Monopoly game.

They do not replace the evaluator's own reasoning.

---

### Supporting Evidence

Every calculation should be considered as one additional source of evidence.

The evaluator should consider:

- the calculation itself;
- its intermediate calculations;
- its final conclusions;
- the supporting Board State;
- the Trade Information;
- the Game Configuration.

The evaluator should understand how the calculation was produced before relying on its conclusions.

---

### Independent Evaluation

The evaluator should independently assess every important calculation.

The evaluator may:

- fully agree;
- partially agree;
- disagree.

Whenever disagreement exists, the evaluator should explain the evidence supporting the alternative conclusion.

---

### Multiple Algorithms

The input may contain evidence generated by multiple Static Algorithms.

Each algorithm should be evaluated independently.

Agreement between algorithms may increase confidence.

Disagreement should not automatically invalidate any algorithm.

The evaluator should analyze the evidence supporting each calculation before reaching a conclusion.

---

### Missing Calculations

Static Algorithms are not required to generate every possible calculation.

The evaluator should use whatever deterministic evidence is available.

Missing calculations should reduce the amount of available evidence, but should not prevent the evaluator from completing the trade evaluation.

---

### Future Compatibility

New calculation types may be introduced in future versions of the Static Algorithm.

The evaluator should consider any new documented calculation type as additional supporting evidence.

Undocumented calculation types should not be interpreted beyond their documented meaning.
